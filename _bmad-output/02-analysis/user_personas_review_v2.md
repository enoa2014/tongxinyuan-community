# 用户画像与系统实现全景回顾 (User Persona & Implementation Review)
**日期**: 2026-01-31
**版本**: v1.0 (Phase 5 Completion)

本文档旨在全面回顾同心源社区支持中心的核心用户群体，并对比《2026产品设计》与当前系统实现的匹配度，明确下一步开发重点。

---

## 1. 公众与潜在捐赠者 (The Public & Donors)
> **画像**: 关心公益的社会人士、企业合作伙伴、寻找求助信息的路人。
> **核心诉求**: 信任感、信息透明、便捷的捐赠/求助指引。

| 功能模块 | 理想状态 (To-Be) | 当前现状 (As-Is) | 差距分析 |
| :--- | :--- | :--- | :--- |
| **门户首页** | 视觉冲击力强，清晰展示四大服务，实时数据看板。 | ✅ **已上线**。包含 Hero Banner, 影响力数据, 新闻动态。 | 基本达成。 |
| **信任建设** | 实时更新的项目动态、财务披露、合规资质。 | ✅ **已上线**。CMS 允许发布新闻动态。 | 需持续运营填充内容。 |
| **捐赠渠道** | 顺畅的捐赠指引 (跳转腾讯公益/支付宝)。 | ✅ **已上线**。独立捐赠页 (`/donate`)。 | 无在线支付功能 (合规限制)，目前状态合规。 |
| **互动性** | 在线留言、活动报名、志愿者招募入口。 | 🚧 **半成品**。仅有静态联系方式和志愿者报名表。 | 缺乏直接的用户反馈渠道。 |

---

## 2. 大病患儿家庭 (Patient Families) - *[重点缺口]*
> **画像**: 异地求医、经济拮据、心理焦虑。需要住宿、物资和心理支持。
> **核心诉求**: “能住哪？” “能领什么？” “申请进度怎么查？”

| 功能模块 | 理想状态 (To-Be) | 当前现状 (As-Is) | 差距分析 |
| :--- | :--- | :--- | :--- |
| **服务发现** | 清晰查找只需的服务（住宿/物资/厨房）。 | ✅ **已上线**。服务介绍页 (`/services`)。 | 信息展示层面已达标。 |
| **申请流程** | 手机端一键申请，**自助查询进度**，无需反复打电话。 | ❌ **缺失**。仅有简单的“咨询表单”，提交后石沉大海。 | **体验断层**。目前只能等人工回访，无法自助。 |
| **个人档案** | 建立电子档案 (实名/病历)，一次建档，多次受益。 | ❌ **缺失**。每次咨询都是独立记录，数据未打通。 | **核心合规缺口**。需要“建档立卡”系统。 |
| **会员中心** | 登录后查看住宿记录、积分、领物记录。 | ❌ **缺失**。无家庭端登录入口。 | 暂无。 |

## 3. 志愿者 (Volunteers) - *[次级缺口]*
> **画像**: 大学生、专业人士。希望贡献时间并获得认可。
> **核心诉求**: 报名便捷、工时记录准确、获得证书/认可。

| 功能模块 | 理想状态 (To-Be) | 当前现状 (As-Is) | 差距分析 |
| :--- | :--- | :--- | :--- |
| **招募报名** | 在线填写技能表，快速响应。 | ✅ **已上线**。志愿者申请流程通畅及后台审批。 | 达标。 |
| **服务记录** | 自动记录服务时长，生成电子证书。 | ❌ **缺失**。后台有记录，但志愿者自己看不到。 | 缺乏正反馈机制。 |
| **任务体系** | 在线抢单/排班 (Task Board)。 | ❌ **缺失**。目前依然依赖微信群吼。 | 运营效率低。 |

## 4. 内部核心团队 (Internal Staff) - *[架构升级中]*
在此次规划中，我们将原有的单一“管理员”拆分为三个角色。

### 4.1 🟢 一线社工 (Social Worker/Operator)
> **职责**: 具体的服务交付、志愿者管理、家庭探访。
> **现状**: 使用超级管理员账号，权限过大，风险高。

| 需求 | 状态 | 改进计划 (RBAC) |
| :--- | :--- | :--- |
| **志愿者运营** | ✅ 可审批/备注 | **授权**: 审批、排班、评价。 **禁权**: 删号/重置密码。 |
| **个案服务** | 🚧 记录缺失 | **新增**: “服务台账” (Service Logs)，记录每次服务细节。 |
| **移动办公** | ❌ 依赖 PC | 需适配手机端管理界面 (Field Mode)。 |

### 4.2 🔵 网站管理员 (Web Admin)
> **职责**: 系统配置、账号安全、内容发布。

| 需求 | 状态 | 改进计划 (RBAC) |
| :--- | :--- | :--- |
| **系统配置** | ✅ 系统设置页 | **授权**: 仅管理员可改全局配置。 |
| **内容管理** | ✅ CMS | 负责全站新闻发布。 |
| **数据隐私** | ⚠️ 可见全部 | **禁权**: 禁止查看受助家庭敏感隐私 (病历/身份证)。 |

### 4.3 🟣 机构管理者 (Manager)
> **职责**: 经营决策、监管审批。

| 需求 | 状态 | 改进计划 (RBAC) |
| :--- | :--- | :--- |
| **数据决策** | 🚧 基础统计 | **增强**: 全局 Dashboard (趋势图/资金流)。 |
| **监管审计** | ❌ 无法追溯 | **新增**: 操作日志审计，查看个案抽查。 |

---

## 总结与下一步
目前系统已完成了**“对外展示”**与**“基础管理”**的 0 到 1。
接下来的核心矛盾在于：**如何让服务过程数字化**（从“记录结果”到“记录过程”）。

**优先级建议**:
1.  **架构升级 (RBAC)**: 先把人分清楚 (社工 vs 管理员)，规避数据风险。
2.  **个案管理 (Case Management)**: 让社工能记“流水账”，满足合规与深度管理。
3.  **服务反馈 (Status Check)**: 最后再做前端查询，让家庭能看到结果。
